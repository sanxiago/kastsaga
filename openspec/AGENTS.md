# Agent Instructions

> **Note:** This file is normally generated by `openspec init` or `openspec update`. The version below is a hand-written stub for projects that have not yet installed the OpenSpec CLI. Run `openspec update` once you have it installed to replace this with the canonical version.

## Reading order for any new task

When asked to plan or build anything in this project, read in this order before responding:

1. `CONSTITUTION.md` — non-negotiable principles. Constraints on every decision.
2. `openspec/project.md` — current tech stack, conventions, vocabulary.
3. `DISCUSSION.md` — unresolved questions. **If a task depends on an unresolved question, stop and surface that, do not improvise.**
4. `openspec/specs/<capability>/spec.md` — current source of truth for the relevant capability.
5. `openspec/changes/` — active proposals that may modify the truth above.

## Workflow

OpenSpec follows a three-phase cycle. The user drives it via slash commands; the AI executes within a phase.

### Phase 1 — Propose

When the user requests a new feature, capability, or modification:

- Create `openspec/changes/<change-name>/`.
- Generate `proposal.md` (why and what), `tasks.md` (implementation checklist), and `specs/<capability>/spec.md` deltas (ADDED / MODIFIED / REMOVED requirements).
- For non-trivial changes, also generate `design.md` describing the technical approach.
- Use kebab-case verb-led names: `add-npc-memory`, `decide-engine`, `update-dialogue-format`.
- Validate with `openspec validate <change-name> --strict` once the CLI is installed.

### Phase 2 — Apply

Only after the user has reviewed and approved the proposal:

- Read `proposal.md` and `tasks.md`.
- Implement tasks sequentially.
- Mark tasks complete (`- [x]`) as work finishes.
- Do not modify the source-of-truth specs in `openspec/specs/` directly during apply — those are updated only at archive time.

### Phase 3 — Archive

When implementation is complete and merged:

- Move the change folder to `openspec/changes/archive/<change-name>/`.
- Merge the change's spec deltas into `openspec/specs/<capability>/spec.md`, applying ADDED / MODIFIED / REMOVED markers.
- The change folder in archive preserves the full history (proposal, design, tasks, deltas) for future reference.

## Naming conventions

- **Capabilities** — verb-noun, kebab-case: `npc-agency`, `world-state`, `player-interaction`. Single purpose. Split if you need "AND".
- **Change IDs** — kebab-case, verb-led: `add-`, `update-`, `remove-`, `refactor-`, `decide-`. Append `-2`, `-3` if a name is taken.
- **References** — link specs as `openspec/specs/npc-agency/spec.md`. Code locations as `path/to/file.ext:42`.

## Constraints from the constitution

These are reminders, not the full text. Always read `CONSTITUTION.md` for the canonical version.

- **P1.** No scripted dialogue trees or fixed FSMs disguised as agency.
- **P3.** World state is authoritative. Never accept an LLM action that contradicts world state.
- **P4.** Every NPC decision point has a latency budget and a fallback action.
- **P5.** All decisions are loggable for replay. Never propose a design that loses determinism.
- **P6.** Guardrails are explicit and inspectable. Never bury safety behind opaque prompt instructions.
- **P9.** Every NPC decision must be inspectable post-hoc. Logs over print statements.

## When to refuse a task

- The task contradicts a constitution principle. Surface this and ask whether to amend the constitution (formal change proposal) or change the task.
- The task depends on a question marked open in `DISCUSSION.md`. Surface which question, recommend a path to resolution.
- The task is out-of-process (e.g., "just write the code"). Ask the user to start a change proposal.

## Tool selection

- **Search by content:** `grep` / `rg`.
- **Search by filename:** `find` / `glob`.
- **Read specific files:** direct read.
- **Run validation:** `openspec validate <change> --strict`.
- **Game engine operations** (once an engine is committed): use the engine's MCP server if available (e.g., Godot AI for Godot 4).
