# NPC Memory (delta: session persistence)

## ADDED Requirements

### Requirement: Memory stores character–NPC interactions only
The system SHALL persist memory entries only for interactions between characters (players or player-linked avatars) and NPCs (including the world-NPC that represents a room or plot owner). Observations, dialogue, goal updates, and environmental actions generated by those interactions SHALL be eligible for storage; incidental noise or interactions exclusively between environment features (without a character or NPC in the loop) SHALL NOT produce lasting memory entries. This keeps memory focused on the relationships and story beats that affect characters while keeping environmental persistence within world-state logs.

#### Scenario: Character causes NPC dislike
- **GIVEN** a character strikes an NPC or breaks an object the NPC values
- **WHEN** the interaction completes
- **THEN** the NPC memory store records an entry tagged `interaction.dislike` with source NPC ID, target character ID, tick, and content describing the incident
- **AND** only because a character was involved; purely environmental changes have no entry in this store

### Requirement: Retention spans correlate with NPC age
The system SHALL assign each NPC an `age` trait and use it to determine retention span. Younger NPCs SHALL have longer retention windows; older NPCs SHALL summarize or forget older entries sooner. Retention policies SHALL be expressible via configurable thresholds (e.g., "retain up to N ticks of interaction memory"), and the policy SHALL be recorded with each entry so auditors can tell if the entry survived due to the NPC's age. Memory entries that fall out of retention SHALL leave a trace (e.g., a summarized placeholder) so the audit trail shows they once existed even though replay may no longer surface them.

#### Scenario: Youthful NPC remembers an insult longer
- **GIVEN** a young NPC with a high retention threshold receives a `dislike` memory about a character from 500 ticks ago
- **WHEN** the NPC's decision loop runs now
- **THEN** the memory entry remains in its context because the age-based policy still includes that tick
- **AND** an older NPC with the same interaction from 500 ticks ago has evicted the entry, leaving only a trace that an interaction once occurred

### Requirement: Situational context layers on stored memory
The system SHALL allow situational prompt context (e.g., current goals, environment cues, NPC traits) to overlay the persistent memory store when constructing the observation/context for the decision backend. This context SHALL be derived from but not duplicate stored entries; context may include summaries of recent interactions, current emotion, or scene metadata while the raw entries live in the memory store for replay.

#### Scenario: Context summarizes recent encounter
- **GIVEN** an NPC has a stored memory entry about a character conspiring in the foyer
- **WHEN** the NPC's decision tick fires while the character is in the same room
- **THEN** the NPC receives a summarized context (e.g., "You still suspect the character") derived from the entry and current scene state
- **AND** the raw entry remains in the store for auditing/replay
